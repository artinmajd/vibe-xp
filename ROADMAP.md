# vibe-xp — Build Roadmap

A gamification web app for a 10-session vibe coding course for teens (ages 13–17). Tracks XP, achievements, and a live leaderboard for 5 teams of 3 students each. Built with Claude Code.

## How to use this document

This roadmap is the source of truth. Work through it phase by phase, top to bottom. **Do not skip ahead.** Each phase ends with a Milestone Checklist — every box must be checked before moving to the next phase. If a milestone fails, fix it before continuing.

Claude Code should reference this file at the start of every session by reading `ROADMAP.md` and `agents.md` together.

---

## Phase 0 — Decisions and accounts (no code yet)

### Stack
- **Framework:** Next.js (App Router, TypeScript, Tailwind)
- **Database + storage + auth:** Supabase (Postgres, Storage, Auth)
- **Hosting:** Vercel
- **Build agent:** Claude Code (local, terminal-based)

### Why this stack
Supabase Auth handles the per-student login requirement natively (email/password or magic link), so we don't build auth from scratch. Supabase Postgres + Storage + Auth all live in one project, one set of env vars. Vercel deploys on git push. Total monthly cost at this scale: $0.

### Accounts to create
1. GitHub (private repo)
2. Vercel (sign in with GitHub)
3. Supabase (sign in with GitHub)

### Local prerequisites
1. Node.js 20.x LTS or higher (`node --version` to verify)
2. Claude Code installed and authenticated
3. `git` configured with your GitHub credentials

### Out of scope (do not build)
- Password reset emails (Supabase handles this if needed later)
- Email notifications
- Native mobile app
- Push notifications
- Per-student leaderboards (XP is team-only by design)

### Milestone 0 — Setup complete
- [ ] GitHub, Vercel, Supabase accounts exist and you can log in to each
- [ ] Node 20+ installed locally
- [ ] Claude Code working in the terminal
- [ ] An empty private GitHub repo named `vibe-xp` exists

## Phase 2 — Database design

### Create the Supabase project
1. Supabase dashboard → New Project → name it `vibe-xp` → region closest to Toronto (likely `us-east-1`) → set and save a strong database password.
2. Wait ~2 minutes for provisioning.
3. In Project Settings → API, copy the Project URL, the `anon` public key, and the `service_role` secret key. Save them — you'll need them in Phase 3.

### Schema (8 tables)

All tables use `uuid` primary keys generated by `gen_random_uuid()` unless noted. All timestamps are `timestamptz` defaulting to `now()`.

**`teams`**
- `id` uuid pk
- `code` text unique not null (the join code, e.g. `NOVA-2847`)
- `name` text not null
- `emoji` text nullable
- `created_at` timestamptz

**`students`**
- `id` uuid pk — must equal `auth.users.id` so Supabase Auth and our profile data join cleanly
- `team_id` uuid fk → teams.id nullable (null = not yet on a team)
- `display_name` text not null
- `email` text not null unique
- `created_at` timestamptz

**`team_members`** (join table, enforces the 3-per-team rule)
- `id` uuid pk
- `team_id` uuid fk → teams.id not null
- `student_id` uuid fk → students.id unique not null (unique = a student can only be on one team)
- `joined_at` timestamptz
- Add a database trigger or check that prevents inserting a 4th row for the same `team_id`. Implementation: a `BEFORE INSERT` trigger that counts existing members and raises an exception if count >= 3.

**`achievements`**
- `id` uuid pk
- `slug` text unique not null (e.g. `letter-counter`)
- `session_number` int not null
- `block_number` int not null
- `title` text not null
- `description` text not null
- `xp` int not null
- `proof_type` text not null (see Phase 2.5)
- `proof_config` jsonb not null default `'{}'`
- `is_secret` boolean default false
- `is_active` boolean default true

**`submissions`**
- `id` uuid pk
- `team_id` uuid fk → teams.id not null
- `student_id` uuid fk → students.id not null (who submitted, for audit only — XP still belongs to the team)
- `achievement_id` uuid fk → achievements.id not null
- `proof_data` jsonb not null default `'{}'`
- `screenshot_url` text nullable
- `status` text not null check in (`auto_approved`, `pending`, `approved`, `rejected`)
- `xp_awarded` int not null default 0
- `submitted_at` timestamptz
- `reviewed_at` timestamptz nullable
- **Unique constraint:** `(team_id, achievement_id)` — a team can only submit each achievement once.

**`sessions`**
- `id` int pk (1 through 10)
- `title` text not null
- `is_active` boolean not null default false
- **Partial unique index:** ensure only one row has `is_active = true` at a time (`CREATE UNIQUE INDEX one_active_session ON sessions (is_active) WHERE is_active = true`).

**`instructor_actions`** (audit log)
- `id` uuid pk
- `instructor_email` text not null (so you know which of you did what)
- `submission_id` uuid fk nullable
- `team_id` uuid fk nullable
- `action` text not null
- `xp_delta` int default 0
- `note` text nullable
- `created_at` timestamptz

**`manual_xp_grants`** (instructor-awarded XP that isn't tied to a submission, e.g. Persistence secret)
- `id` uuid pk
- `team_id` uuid fk not null
- `xp` int not null
- `reason` text not null
- `granted_by` text not null (instructor email)
- `created_at` timestamptz

### Proof types (the extensibility pattern)

Each achievement declares its own validation contract via `proof_type` + `proof_config`. New achievement types in later sessions never require schema changes — only new validator functions.

- `screenshot` — config: `{}`. Validates that `screenshot_url` is non-null.
- `url` — config: `{}`. Validates that `proof_data.url` matches an http(s) regex.
- `text` — config: `{ "min_words": 200 }`. Validates word count of `proof_data.text`.
- `checklist` — config: `{ "items": ["...", "..."] }`. Validates all items in `proof_data.checked` array.
- `fields` — config: `{ "fields": ["Label 1", "Label 2"] }`. Validates all fields in `proof_data.values` are non-empty.
- `code_entry` — config: `{}`. Validates that `proof_data.code` matches another team's join code.
- `composite` — config: `{ "require": ["screenshot", "checklist"], "items": [...] }`. Runs multiple validators.
- `instructor_flag` — config: `{ "description": "..." }`. Always lands in `pending` queue.

### Storage bucket
1. Supabase dashboard → Storage → New bucket called `screenshots`, public read enabled, max file size 5 MB.
2. Allowed mime types: `image/png, image/jpeg, image/webp`.

### RLS strategy for v1
- Keep RLS **off** for `teams`, `students`, `team_members`, `achievements`, `submissions`, `sessions`, `instructor_actions`, `manual_xp_grants`.
- All writes go through Next.js API routes using the service-role key (server-only).
- The browser never holds the service-role key. Only the `anon` key is used for Supabase Auth login flows.
- This is a deliberate v1 simplification. Add RLS post-launch if needed.

## Phase 4 — Authentication

This phase adds per-student login. Each student gets their own account; teams form via a separate join flow in Phase 6.

### Tasks
1. In Supabase Auth settings: enable Email provider, **disable email confirmation** for v1 (teens won't reliably click links — turn back on later if abuse appears). Set Site URL to `http://localhost:3000` for now.
2. Create `app/(auth)/signup/page.tsx`: client component with email + password + display name fields. On submit, calls Supabase Auth `signUp`, then POSTs to `/api/students/create-profile` to insert the matching `students` row.
3. Create `app/(auth)/login/page.tsx`: email + password, calls `signInWithPassword`.
4. Create `app/api/students/create-profile/route.ts`: server-side, takes `{ user_id, email, display_name }`, inserts into `students` with `id = user_id` and `team_id = null`.
5. Create middleware at the project root (`middleware.ts`) that refreshes the auth session on every request using `@supabase/ssr`.
6. Create `app/(auth)/logout/route.ts` that signs the user out and redirects to `/login`.
7. Create a route guard helper: `lib/require-auth.ts` exporting a function that reads the session server-side and redirects to `/login` if missing.

### Design notes
- Use email format `firstname.lastname@class.local` for v1 if students don't have email addresses. Disabling email confirmation makes this work.
- Display name shows up in the team page and submission audit log — keep it short.
- No password complexity rules in v1. These are teens, not banks.

## Phase 6 — Team formation flow

This is where students log in, create or join teams, and the team_members table fills up.

### Tasks
1. Create `app/dashboard/page.tsx` — the first page a logged-in user sees. If `students.team_id` is null, redirect to `/team-setup`. Otherwise show the team home (Phase 7).
2. Create `app/team-setup/page.tsx` with two paths: "Create a new team" and "Join an existing team."
3. **Create team flow:**
   - Input: team name (required), emoji (optional)
   - On submit, POST to `/api/teams/create` which: generates a unique join code (format `WORD-####`, e.g. `NOVA-2847`), inserts into `teams`, inserts the creator into `team_members`, updates the creator's `students.team_id`
   - Returns the join code and redirects to `/dashboard`
4. **Join team flow:**
   - Input: join code
   - On submit, POST to `/api/teams/join` which: looks up the team by code, checks current member count via the trigger (will fail loudly if team is full), inserts into `team_members`, updates `students.team_id`
   - On full team, return a clear error message: "This team already has 3 members."
5. **Constraints to enforce in the API routes:**
   - A student with `team_id != null` cannot create or join another team. Return error.
   - A team name must be unique per cohort (add a unique constraint on `teams.name` if not already there).
   - Limit total teams per active session to 5 (the spec requires exactly 5 teams of 3). Add an instructor override in case a real cohort has different numbers — controlled via env var `MAX_TEAMS=5`.
6. Create `app/api/teams/leave/route.ts` for the rare case a student needs to leave a team (clears their `team_id` and removes the `team_members` row). Do NOT delete submissions tied to them — they belong to the team historically.

### Auto-awarded achievements on team formation
- When a team reaches 3 members, automatically insert an `auto_approved` submission for the `team-names` achievement (+2 XP per spec).
- The `team-ideas` achievement gets a separate form on the team home page (Phase 7) where each member submits their idea — when all 3 have submitted, award +2 XP.

### Milestone 6 — Teams forming
- [ ] A logged-in student with no team is redirected to `/team-setup`
- [ ] Creating a team generates a unique code, adds the creator as the first member, sets their `team_id`
- [ ] A second student can join with the code and becomes member #2
- [ ] A fourth student trying to join the same team gets a clear error
- [ ] A student already on a team cannot create or join another
- [ ] When the third member joins, the `team-names` submission appears and +2 XP shows on the leaderboard
- [ ] When all 3 members submit their app idea, +2 XP is awarded
- [ ] You can't create a 6th team if `MAX_TEAMS=5`

## Phase 8 — Validation logic

### Tasks
1. Create `lib/validators.ts` with one function per proof type, each returning `{ valid: boolean, reason?: string }`:
   - `validateScreenshot(proofData, screenshotUrl)`
   - `validateUrl(proofData)` — regex check for http(s) URL
   - `validateText(proofData, config)` — word count vs `config.min_words`
   - `validateChecklist(proofData, config)` — every item in `config.items` must appear in `proofData.checked`
   - `validateFields(proofData, config)` — every field in `config.fields` must have a non-empty value in `proofData.values`
   - `validateCodeEntry(proofData, ownTeamId, supabase)` — async, looks up the entered code, must exist and not equal `ownTeamId`
   - `validateComposite(proofData, screenshotUrl, config, ...)` — runs each required validator
2. Create `lib/dispatch-validator.ts` exporting `runValidator(achievement, proofData, screenshotUrl, teamId, supabase)` that switches on `achievement.proof_type` and calls the right function.
3. The `/api/submit` endpoint uses `runValidator`. If valid → `auto_approved`, full XP. If invalid → `pending`, 0 XP, with the `reason` saved in `proof_data._validation_reason` so the instructor sees why on review.
4. **Total XP computation lives in one place.** Create `lib/team-xp.ts` exporting `getTeamXP(teamId)` that returns `{ sessionXp, totalXp, level }`. It sums `xp_awarded` from `submissions` where status is `auto_approved` or `approved`, plus `xp` from `manual_xp_grants`. Never store total XP anywhere.
5. Create `lib/levels.ts` with `xpToLevel(xp)` returning `{ level, name, nextThreshold, xpToNext }`. Thresholds: Builder 0, Creator 50, Inventor 100, Engineer 150, Architect 220, Founder 300, AI Master Builder 400.

### Milestone 8 — Scoring is correct
- [ ] All seven validators have unit tests in `__tests__/validators.test.ts` (use `vitest` or Node's built-in test runner) — at minimum one passing case and one failing case each
- [ ] `getTeamXP` returns correct totals across mixed `auto_approved`, `approved`, and `manual_xp_grants` records
- [ ] `xpToLevel(75)` returns `{ level: 2, name: "Creator", nextThreshold: 100, xpToNext: 25 }`

## Phase 10 — Secret achievements

### Tasks
1. Create `lib/check-secrets.ts` exporting `checkSecretAchievements(teamId, supabase)`. It runs every time `/api/submit` succeeds.
2. Implement the four Session 1 secrets:
   - **`skeptic`**: count of approved submissions for `letter-counter`, `double-down`, `hallucination-hunter`, `contradiction-machine` ≥ 3 → auto-award +5 XP
   - **`persistence`**: hard to auto-detect → defer as instructor-awarded via `manual_xp_grants`
   - **`collaborator`**: count of approved `neighbor-assist` submissions where this team's code was the one entered ≥ 2 → auto-award +5 XP
   - **`builders-apprentice`**: count of distinct `student_id` values in this team's approved submissions ≥ 3 (i.e. all 3 members have submitted something) → auto-award +5 XP
3. Each auto-award inserts a submission row with `status = "auto_approved"` and links it to the secret achievement.
4. The team dashboard shows a one-time celebration toast for newly unlocked secrets. After dismissal, the secret appears in the team's earned-achievements list.
5. Locked secrets are never shown anywhere — not on the dashboard, not in the achievement list. They appear only after being earned.

### Milestone 10 — Secrets work
- [ ] Submitting the third "hallucination-type" achievement triggers `skeptic` automatically
- [ ] Three different team members each submitting one achievement triggers `builders-apprentice`
- [ ] A team being credited via `neighbor-assist` twice triggers `collaborator`
- [ ] An instructor can manually grant `persistence` via the instructor dashboard (Phase 11)
- [ ] Unlocked secrets show up in the team's earned list; locked ones are invisible

## Phase 13 — Pre-class preparation

### Tasks
1. **Reset script.** Create `scripts/reset-cohort.ts` that truncates `submissions`, `team_members`, `manual_xp_grants`, `instructor_actions`, sets all `students.team_id` to null, deletes all `teams`, but leaves `achievements`, `sessions`, and `auth.users` intact. Add a confirmation prompt.
2. **Sign-in cards.** For each student, generate a printed card with their pre-created login email and a temporary password. (Alternative: have them sign up on arrival, but pre-creating is faster.) Create `scripts/generate-students.ts` that takes a list of names and pre-creates `auth.users` + `students` rows.
3. **Team formation guidance.** Decide before class whether students pick their own teams or are pre-assigned. Either is fine — the app supports both.
4. **Dress rehearsal.** Sit with your co-instructor at two laptops. Pretend to be students. Sign up, form a team, submit one of every achievement type. Time it. Aim for under 30 seconds per submission.
5. **Load test.** Open 5 browser tabs and submit at the same time. Supabase free tier handles this easily — but verify.
6. **Backup plan.** Have a Google Sheet ready as a manual XP tracker in case the app goes down mid-session. You probably won't need it; have it anyway.
7. **Venue check.** Wi-Fi reaches every table, projector connects to your laptop, phone hotspots work as backup.

## Phase 14 — Sessions 2–10 extension pattern

This is documentation, not a build phase. Use this pattern when adding achievements for later sessions.

### Adding a new achievement
1. Add a row to the achievements array in `scripts/seed.ts` with the right `session_number`, `slug`, `proof_type`, and `proof_config`.
2. If the proof type is one of the existing 8, you're done — re-run the seed and the achievement appears.
3. If it's a new proof type:
   - Add a validator function in `lib/validators.ts`
   - Add a case to `runValidator` in `lib/dispatch-validator.ts`
   - Add a form component in `components/proof-forms/`
   - Add a case to the renderer switch in `app/dashboard/achievement/[slug]/page.tsx`
   - About 30 minutes of work

### Between sessions
1. Log into `/instructor`
2. Flip the active session to the next number
3. The team dashboard now shows that session's achievements
4. Previous sessions' achievements remain visible in the team's earned list — never archive

### For Session 10 presentations
1. Switch leaderboard view to "Total XP" via `?view=total`
2. Project it during closing ceremony
3. Use the `instructor_actions` log to surface "biggest comeback team," "most secret achievements," etc., as final reflections
